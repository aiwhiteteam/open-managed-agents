# Compatibility Matrix

This MVP targets Claude Managed Agents-shaped wire and lifecycle compatibility, not identical Claude behavior.

| Area | Status | Notes |
| --- | --- | --- |
| `/v1` URL shape | Implemented | Matches Anthropic-style stable root path. |
| Native beta header | Implemented | `open-managed-agents-beta: open-managed-agents-2026-04-01` is accepted without Anthropic headers. |
| Claude compatibility headers | Implemented | `anthropic-beta: managed-agents-2026-04-01` is accepted; `anthropic-version: 2023-06-01` is required when using `anthropic-beta`. |
| Agent create/list/retrieve | Implemented | Flexible JSON config fields, non-empty name/model validation, model `speed` validation, OpenAI-compatible model alias normalization, and official SDK `retrieve(version=...)`. |
| Agent update version guard | Implemented | Requires current `version`; stale writes return 409. |
| Agent no-op update detection | Implemented | Existing version is returned. |
| Metadata bags | Implemented | Agents, sessions, environments, vaults, credentials, memory stores, deployments, and user profiles enforce 16 keys, 64-character keys, and 512-character values. Create requires string values; update merges by key and deletes with `null` or empty string where supported by the SDK request shape. |
| Agent tool validation | Implemented | Enforces the 128-tool aggregate limit, custom tool name uniqueness/format, custom description limits, MCP server max count, and MCP toolset/server reference consistency. |
| Multiagent roster validation | Implemented | Coordinator rosters resolve unversioned agent references to the referenced agent's active version at create/update time, support SDK string entries, enforce 1-20 distinct entries, at most one `self`, positive versions, and depth limit 1. |
| MCP server/toolset declaration validation | Implemented | Rejects unreferenced MCP servers and dangling `mcp_toolset` references. Runtime context matches session vault credentials by MCP URL and emits non-terminal missing-auth session errors; real MCP connection/OAuth refresh is still partial. |
| Agent archive | Implemented | Blocks new sessions; existing sessions can continue. |
| Environment CRUD | Implemented | Create/retrieve/update/list/archive/delete pass official SDK strict validation for the covered lifecycle, including self-hosted `scope` and rejection of `scope` on cloud environments. Sandbox backend mapping is not complete. |
| Session create/list/retrieve/update/archive/delete | Implemented | Covered lifecycle passes official SDK strict validation. Sessions pin agent version at creation. Running/rescheduling sessions reject update/archive/delete. |
| Session state machine | Partial | Core states are `idle`, `running`, `rescheduling`, and `terminated`; `requires_action` is represented as an idle `stop_reason`. Transient runtime failures enter `rescheduling` with retry windows and capped attempts, emitted as SDK-compatible `session.status_rescheduled` events. Durable retry execution is still TODO. |
| Session-local agent update | Partial | `agent.tools` and `agent.mcp_servers` replacement passes official SDK strict validation and does not mutate the persisted agent version. Full runtime semantics are TODO. |
| Session events append/list | Implemented | Append-only event log with monotonic `seq`. Basic official SDK strict validation passes for send/list, including `system.message` one-per-request, final-event, immediate-predecessor validation. OpenAI Agents SDK run-item stream mapping covers messages, tool calls/results, MCP approvals, reasoning as `agent.thinking`, handoffs, and agent updates; raw response deltas and the exact full Claude event union are still partial. |
| Session resources | Partial | Official SDK strict validation passes for session-create resource union (`file`, `github_repository`, `memory_store`), file add, GitHub token rotation, retrieve/list/delete, memory-store snapshots, and GitHub token redaction. Uploaded file mounts create session-scoped file resources and object-storage copies. Memory-store resources enforce the official 8 stores per session limit and cannot be added or removed after creation. Real filesystem mount semantics and session-produced file references are still TODO. |
| Requires action continuation | Partial | Local/runtime contract supports `agent.custom_tool_use` + `user.custom_tool_result`, self-hosted `agent.tool_use` + `user.tool_result`, and tool confirmation through `user.tool_confirmation`. Full OpenAI Agents SDK HITL persistence is TODO. |
| Outcomes | Partial | `user.define_outcome` starts work, emits SDK-compatible `span.outcome_evaluation_end`, and records session `outcome_evaluations` with rubric and max-iteration metadata. Real grader LLM loops are TODO. |
| Multiagent threads | Partial | Coordinator sessions create delegated `session_thread` resources from pinned rosters with separate agent snapshots and parent primary thread references. Primary thread event listing includes unassigned session events, while delegated thread event listing only includes explicitly routed events. Full multiagent execution/event routing is TODO. |
| SSE stream | Implemented | DB polling-based replay and live polling. |
| OpenAI Agents SDK runtime | Partial | Uses SDK with OpenAI by default. Deterministic local runtime is explicit test-only configuration. Advanced OpenAI-compatible providers can be registered through `OMA_OPENAI_COMPATIBLE_PROVIDERS`, and API-level session turns have mocked OpenAI-compatible runtime coverage. SDK `RunState.to_json()` snapshots are persisted when available; reconstructing and resuming through `RunState.from_json()` plus wire-level mocked OpenAI SDK HTTP endpoint coverage are still TODO. |
| SandboxAgent mapping | Partial | Environment sandbox config maps to OpenAI Agents SDK `SandboxAgent` + `RunConfig.sandbox` for explicit `unix_local` tests. Networking/package/resource policy config validation and runtime sandbox-state summaries are covered locally. Production cloud/self-hosted sandbox lifecycle and real policy enforcement are TODO. |
| Exact Anthropic Python SDK contract tests | Partial | `tests/contract/test_anthropic_sdk_contract.py` points the official Anthropic Python SDK at this ASGI app with strict response validation. Current coverage includes SDK surface, agents, agent versions, environments, sessions, session events/resources/threads, SDK SSE decoder/stream event union parsing, files, skills, skill versions, vaults, credentials, memory stores, memories, memory versions including `api_key_id`/`session_id`/`view` list filters, deployments, deployment runs, user profiles, representative SDK cursor pagination, route-specific max-limit clamping, session list filters, and key resource list filters. Exhaustive pagination edge cases and production runtime semantics are still TODO. |
| Workspace isolation | Implemented | API-key-to-workspace scoping is covered for agents, environments, files, skills, vaults, credentials, memory stores, memories, deployments, deployment runs, and user profiles. Hosted org/RBAC semantics remain private-layer work. |
| Skills API | Partial | Create/list/retrieve/delete plus versions and zip content download. Basic official SDK strict validation passes for skill and skill version lifecycle. Custom skill versions use official-compatible epoch-microsecond identifiers, and agent skill refs use official `custom`/`anthropic` union shapes with custom versions validated as concrete versions or `latest`. S3-compatible object storage is supported through `S3_*` settings. Uploads require one top-level directory, safe relative file paths, `SKILL.md` frontmatter with `name` and `description`, and local EICAR content-scan checks; exact Claude archive/runtime semantics are TODO. |
| Files API | Partial | Upload/list/retrieve/download/delete with S3-compatible object storage. Basic official SDK strict response validation passes, including `after_id`/`before_id` pagination and session `scope_id` list filtering. Uploads deduplicate by content hash within a workspace, direct/staged uploads pass local EICAR content-scan checks, staged upload completion is workspace-scoped, session file resource mount response shape passes, and uploaded file mounts create session-scoped file resources and object-storage copies with absolute mount-path validation and the official 100 file resources per session limit. Real filesystem mount semantics, production malware scanning, and lifecycle policies are still TODO. |
| Vaults and credentials | Partial | Basic lifecycle response shapes pass official SDK strict validation, including `mcp_oauth`, `static_bearer`, and `environment_variable` credential auth unions. API responses redact credential secret values including OAuth refresh secrets, environment-variable partial auth updates preserve immutable `secret_name`, archive/delete purges persisted secret values from the MVP generic table, validation persists last-validation metadata, and sessions validate workspace-scoped `vault_ids`. Secure secret storage, real OAuth enrollment/refresh/probing, runtime resolution, permission enforcement, and webhook emission are TODO. |
| Webhook helpers | Partial | Standard Webhooks-compatible signing, verification, freshness checks, and event-envelope unwrap helpers have local coverage. Endpoint registration, event delivery, retries, idempotency, failure disabling, and emitted Managed Agents webhook event coverage are TODO. |
| Memory stores | Partial | Postgres-backed path memories with SDK-compatible store name/description limits, slash-prefixed path validation, indexed exact/prefix path lookup, `depth` rollups as `memory_prefix` items, unique path, content hashes, optimistic version checks, version history with actor attribution, memory-version `api_key_id`/`session_id`/`operation`/`view` filtering, deleted-version survival, live-head redact protection, archived-store read-only behavior, official 100KB memory and 2000 memories per store limits, 8 mounted stores per session, runtime context injection for mounted stores, and official SDK strict response validation. Full memory tools and semantic/vector indexes are TODO. |
| Deployments and runs | Partial | Basic lifecycle response shapes pass official SDK strict validation, including cron schedule shape, computed upcoming run timestamps, deployment `system.message` initial-event ordering, deployment `resources`/`vault_ids` limits, deployment list `agent_id`/`status` filters with `status` vs `include_archived` validation, deployment-run `has_error` semantics, pause/unpause with manual paused reasons, manual runs while paused, terminal archive semantics, primary-agent archive auto-archive/no-run behavior, failed session-creation run records, deployment-run session linkage, vault validation for deployment-created sessions, and an importable idempotent due-schedule tick. Production scheduler/retry worker semantics are TODO. |
| User profiles | Partial | Basic lifecycle and enrollment URL response shapes pass official SDK strict validation. Enrollment URLs use generated tokens with hashed persisted state. Real identity binding, hosted verification, trust grants, and access policy are TODO. |
| Self-hosted environment work | Partial | Optional hidden/advanced path with Postgres-backed work items, official SDK strict response validation for poll/list/retrieve/update/ack/heartbeat/stats/stop, optional worker token auth, poll lease, HTTP/CLI lease owner enforcement, expired-lease recovery, retry-at backoff gates, and `oma-worker`. Durable retry execution, hosted worker identity/RBAC, and external worker deployment are TODO. |
